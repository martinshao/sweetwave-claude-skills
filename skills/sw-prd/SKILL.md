# SweetWave PRD

用户输入：

```txt
$ARGUMENTS
```

## 输入

优先读取：

- `docs/sweetwave/00-brief/BRIEF.md`
- 用户本次输入的补充信息
- 用户输入中指向的需求文档或目录
- 如存在，读取当前已有 `docs/sweetwave/01-prd/PRD.md`

如果用户提供的是目录路径：

- 优先读取该目录下的 `docs/` 子目录。
- 读取其中 `.md`、`.txt` 等文本需求文档并综合分析。
- 如目录或 `docs/` 不存在、为空，明确提示用户补充需求源。

如果用户输入以 `--change` 开头：

- 进入需求变更模式。
- 必须先读取已有 `docs/sweetwave/01-prd/PRD.md`。
- 将变更内容合并进 PRD，并在需求版本、功能需求、验收标准、风险与待确认问题中体现变化。
- 不要丢失已有仍然有效的需求。

## 输出文件

创建或更新：

```txt
docs/sweetwave/01-prd/PRD.md
```

## 工作流程

1. 判断模式：
   - 普通模式：根据产品简报、用户输入或需求文档生成 PRD。
   - 变更模式：根据 `--change` 后的说明更新已有 PRD。
2. 收集需求源：
   - 阅读 `BRIEF.md`、用户补充信息、用户指定的文档或目录。
   - 如果有多个需求文件，全部阅读后合并分析。
3. 获取项目上下文：
   - 从 `package.json`、workspace 配置、目录名等线索提取项目名。
   - 扫描项目根目录和常见配置，判断项目类型或架构形态，例如单体应用、前后端分离、monorepo、移动端、CLI、库、后端服务等。
   - 如存在 `CLAUDE.md`，读取长期有效的项目规则。
4. 分析需求：
   - 提取产品目标、目标用户、核心场景、功能需求、非功能需求、约束、依赖和验收标准。
   - 将功能需求编号为 `[F-001]`、`[F-002]`。
   - 将验收标准编号为 `[AC-001]`、`[AC-002]`，并关联到对应功能需求。
5. 开放问题处理：
   - 对低风险缺口，可以做合理假设，但必须写入「风险与假设」。
   - 如果存在高风险歧义，必须先向用户确认，不要强行生成最终 PRD：
     - 核心业务规则矛盾或不完整。
     - 权限、支付、隐私、安全、数据删除、敏感操作不明确。
     - 目标平台、兼容性、第三方服务或合规要求缺失。
     - 多种产品方向都会显著影响范围和验收标准。
6. 生成或更新 PRD：
   - 保留已有 PRD 中仍然有效的内容。
   - 删除或标注被变更取代的内容，避免新旧需求互相冲突。
   - 确保每条核心需求都可测试、可验收。
7. 输出总结：
   - 说明 PRD 是新建还是更新。
   - 概述项目名、项目类型、核心功能数、关键待确认问题。

## 规则

- 不要写代码。
- 不要做具体技术架构。
- 不要在 PRD 阶段生成设计文档、开发规格或任务清单。
- 每条需求都要尽量可测试、可验收。
- 功能需求和验收标准必须有稳定编号，方便后续 `/sw-design`、`/sw-spec`、`/sw-task` 引用。
- 如果需求很大，必须建议拆成多个独立 feature 或阶段，避免一个 PRD 覆盖过大的交付范围。
- 明确区分：事实、用户明确要求、合理假设、待确认问题。
- 输出语言使用中文。
